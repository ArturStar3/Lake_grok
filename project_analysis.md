# Анализ проекта lake_grok (infolake)

**Дата анализа:** 2026-06-18  
**Анализатор:** Grok (на основе project_context.md + targeted чтение исходников)

## 2. Дублирование логики (logic duplication)

**Файл:** `backend/api/serializers.py:244-263` и `backend/api/views.py:81-101`  
**Строка(ы):** см. выше  
**Описание проблемы:** Практически идентичный код создания `TargetAction` из `action_type_id` + `radius`: цикл + `ActionType.objects.get` + `create`. Один в `TargetCreateSerializer.create`, второй в `TargetViewSet.update`.  
**Предлагаемое решение:** Вынести в utility-функцию (например `create_target_actions(target, actions_data)`) или сделать полностью writable nested serializer для actions.  
**Ожидаемый эффект:** DRY, проще поддержка, меньше ошибок при изменениях.

**Предлагаемое решение:** Вынести общий фильтр в утилиту `filterTargets(objects, {title, types, countries?})`.  
**Ожидаемый эффект:** Меньше кода, легче добавлять фильтры, меньше рассинхронизации.

**Файл:** `backend/api/views.py:350-376` (EventViewSet.get_queryset)  
**Строка(ы):** 350-376  
**Описание проблемы:** Дублирующийся шаблон построения Q для date/time фильтров (много if/elif с почти одинаковым кодом для date и time).  
**Предлагаемое решение:** Ввести helper `build_date_range_q(...)` и `build_time_range_q(...)`.  
**Ожидаемый эффект:** Читаемость, меньше багов при правках фильтров.

**Файл:** `frontend/src/components/MapComponent/` (маркерная кластеризация)  
**Строка(ы):** `markerClusteringUtils.js`, `NonFlagMarkerUtils.jsx`, `MapUtils.jsx`

## 3. Потенциальные ошибки и баги (potential errors)

**Файл:** `backend/formular/models.py`  
**Строка(ы):** 380  
**Описание проблемы:** Опечатка `related_name='contries'` у FK `Target.country` (вместо 'countries' или 'targets'). Обратный доступ `country.contries` сломан (должен работать как `country.target_set` или переименованный).  
**Предлагаемое решение:** Переименовать в `related_name='targets'`, создать миграцию.  
**Ожидаемый эффект:** Корректные reverse relations, отсутствие путаницы.

**Файл:** `backend/formular/admin.py`  
**Строка(ы):** 45

**Предлагаемое решение:** Исправить на `list_filter`.  
**Ожидаемый эффект:** Фильтр заработает.

**Файл:** `backend/api/views.py:87` (и аналог в serializers)  
**Строка(ы):** TargetViewSet.update + TargetCreateSerializer.create  
**Описание проблемы:** Удаление + создание actions происходит вне транзакции. При ошибке в середине цикла возможно частичное состояние (старые actions удалены, новые не все созданы).  
**Предлагаемое решение:** Обернуть в `transaction.atomic()`.  
**Ожидаемый эффект:** Целостность данных при обновлении Target.

**Предлагаемое решение:** Раскомментировать unique + миграция (или добавить unique_together/validate).  
**Ожидаемый эффект:** Целостность справочника стран.

**Файл:** `backend/infolake/settings.py:34,36,149`  
**Строка(ы):** 31 (SECRET), 34 (DEBUG), 36 (ALLOWED_HOSTS), 149 (CORS)

**Ожидаемый эффект:** Безопасность при деплое.

## 4. Узкие места производительности (performance bottlenecks)

**Файл:** `backend/api/views.py:55-57` (TargetViewSet)  
**Строка(ы):** 55  
**Описание проблемы:** `select_related('country', 'marker', 'type')`, но отсутствует `prefetch_related('actions')`. При этом `TargetSerializer` вкладывает `actions` (many). При запросе списка целей возникает N+1 (по одному запросу на actions для каждого target).  
**Предлагаемое решение:** Добавить `.prefetch_related( Prefetch('actions', queryset=TargetAction.objects.select_related('action_type')) )`.  
**Ожидаемый эффект:** Значительное снижение количества запросов при загрузке 100–1000+ объектов.

**Файл:** `backend/api/views.py:156-168` (CountryInfoView)  
**Строка(ы):** 161, 166  
**Описание проблемы:** Простой `get` + `filter` без select/prefetch. Сериализатор вкладывает section. При множестве записей — потенциальный N+1.  
**Предлагаемое решение:** Использовать `select_related` + prefetch при необходимости.  
**Ожидаемый эффект:** Быстрее ответы /country/<iso>/.

**Файл:** `backend/formular/admin.py:193-235` (MarkerAdmin.svg_thumbnail)  
**Строка(ы):** 198-229  
**Описание проблемы:** На каждой строке списка маркеров выполняется синхронное чтение файла SVG + несколько re.sub + regex. При 100+ маркерах админка сильно тормозит.  
**Предлагаемое решение:** Кэшировать обработанный SVG (в памяти или поле в БД), либо вынести генерацию превью в отдельное действие/сигнал.  
**Ожидаемый эффект:** Админка маркеров становится отзывчивой.

**Файл:** `backend/api/views.py` + DRF (глобально)  
**Строка(ы):** нет пагинации  
**Описание проблемы:** Отсутствует настройка пагинации в REST_FRAMEWORK. Полный список targets/events загружается целиком (могут быть тысячи).  
**Предлагаемое решение:** Добавить в settings:  
```python
REST_FRAMEWORK = {
    'DEFAULT_PAGINATION_CLASS': 'rest_framework.pagination.LimitOffsetPagination',
    'PAGE_SIZE': 100,
}
```  
Обновить фронт на работу с пагинированными ответами (или оставить для справочников).  
**Ожидаемый эффект:** Защита от OOM и долгой загрузки на больших объёмах данных.

**Файл:** `frontend/src/components/Formular/Formular.jsx` + utils  
**Строка(ы):** fetchData (полная загрузка), filteredObjects, findAllIntersections (используется в useMemo на visibleForIntersections)  
**Описание проблемы:** Всё происходит на клиенте: загрузка полного датасета, фильтры, клиентская геометрия пересечений окружностей (вероятно O(n²) для выбранных), кластеризация при зуме. При 500–1000 объектах — подтормаживания и высокое потребление CPU/памяти браузера.  
**Предлагаемое решение:** Добавить debounce на фильтры/изменения selected, рассмотреть серверную фильтрацию для событий/объектов, virtualize таблицу, кэшировать результаты пересечений, вынести тяжёлые расчёты в web-worker (или ограничить количество одновременно обрабатываемых зон).  
**Ожидаемый эффект:** Плавная работа даже на больших выборках.

**Файл:** `frontend/src/components/MapComponent/NonFlagMarkerUtils.jsx` + clustering  
**Строка(ы):** несколько useEffect + setZoom на zoomend + пересчёты иконок  
**Описание проблемы:** Частые пересчёты кластеров + загрузка/парсинг SVG при зуме/выборе.  
**Предлагаемое решение:** Усилить мемоизацию (pathsKey уже есть), добавить throttle, кэшировать обработанные иконки глобально.  
**Ожидаемый эффект:** Меньше фризов карты.

## 5. Нарушения лучших практик Python / Django (+ React)

**Файл:** `backend/api/views.py` (и все ViewSet)  
**Строка(ы):** 54, 109, 116, ... (17 мест)  
**Описание проблемы:** `permission_classes = [AllowAny]` абсолютно везде. Нет аутентификации/авторизации.  
**Предлагаемое решение:** Ввести базовый `IsAuthenticated`, разделить ReadOnly и write permissions, подготовить JWT или session auth.  
**Ожидаемый эффект:** Соответствие best practices, готовность к продакшену.

**Файл:** `backend/infolake/settings.py`  
**Строка(ы):** 31, 34, 36, 149 + отсутствие REST_FRAMEWORK настроек  
**Описание проблемы:** Отсутствие production-ready конфигурации, нет разделения dev/prod, нет пагинации, CORS открыт.  
**Предлагаемое решение:** Использовать `django-environ` более полно, ввести `infolake/settings/` пакет с base/dev/prod, включить security middleware настройки.  
**Ожидаемый эффект:** Безопасный и масштабируемый деплой.

**Файл:** `backend/api/serializers.py:138-149` (MilitaryBranchSerializer)  
**Строка(ы):** 141  
**Описание проблемы:** `countries = serializers.PrimaryKeyRelatedField(many=True, read_only=True)` — возвращает только PK. Фронтенд (ObjectsTable, модалы) вынужден дополнительно сопоставлять.  
**Предлагаемое решение:** Использовать вложенный простой сериализатор или `SlugRelatedField`, либо prefetch + включить нужные поля.  
**Ожидаемый эффект:** Меньше работы на фронте, полные данные в одном запросе.

**Файл:** `frontend/src/components/Formular/Formular.jsx` (и другие)  
**Строка(ы):** 69 (комментарий `// Это удалить если будет жопа`), множество useEffect/useMemo без чёткой группировки  
**Описание проблемы:** "Магические" комментарии, очень большой компонент-оркестратор (70+ строк только useState + много эффектов). Нарушение single responsibility.  
**Предлагаемое решение:** Вынести часть состояния (action zones, events filters) в кастомные хуки или Context. Удалить debug-комментарии.  
**Ожидаемый эффект:** Поддерживаемый код.

**Файл:** `frontend/src/components/MapComponent/NonFlagMarkerUtils.jsx:6-7`  
**Строка(ы):** 6-7  
**Описание проблемы:** Двойной импорт из одного модуля (`enrichSvg` и `getViewBoxSize` отдельными строками).  
**Предлагаемое решение:** Объединить в один импорт.  
**Ожидаемый эффект:** Чистота.

**Общие (React):**  
- Отсутствие ключей в некоторых списках (нужно проверить рендеры таблиц/групп).  
- Нет обработки ошибок на уровне ErrorBoundary.  
- Много console.* в production-коде (хотя и в catch).

## Сильные стороны проекта (что сделано хорошо)

- Хорошее использование `select_related` / `prefetch_related` + Prefetch в админке и части ViewSet.
- Индексы на моделях (много `indexes`).
- Кастомная админка с SVG-превью, inline, list_editable.
- Много `useMemo`/`useCallback` на фронте для фильтров и кластеризации.
- Полностью подготовленный оффлайн-режим Docker (image + pull_policy: never + комментарии).
- Чёткая структура MilitaryBranch + grouping в ObjectsTable (недавняя фича).
- Использование Leaflet + TileServer GL для полностью оффлайн карт.
- Отдельные утилиты геометрии (circleIntersection).

## Рекомендации (приоритетные)

1. Исправить критичные баги (contries, list_filer, транзакции).
2. Добавить пагинацию + prefetch 'actions' (самый большой выигрыш по perf).
3. Вычистить мёртвый код (App.jsx, data/objects.js, animation?).
4. Рефакторинг дублирования (actions create, фильтры).
5. Ввести разделение settings + базовую авторизацию перед любым прод-деплоем.
6. Добавить базовые тесты (хотя бы smoke для ViewSet и ключевых утилит).
7. Рассмотреть вынос тяжёлой клиентской геометрии/кластеризации в worker или сервер при росте данных.
8. Документировать legacy-поля (action_radius, animation) или удалить.
